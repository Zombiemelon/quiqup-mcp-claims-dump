# Quiqup MCP вЂ” Full Frontend API Coverage

## What This Is

Quiqup MCP is a Next.js-hosted Model Context Protocol server that exposes Quiqup's operational APIs to AI agents (Claude, ChatGPT, internal Mastra agents). The current scope is to grow it from its ~32 existing tools into a **complete mirror of every backend endpoint consumed by the Quiqdash v3 frontend** вЂ” so that anything a human operator can do through the Quiqdash UI, an agent can do through the MCP.

## Core Value

Every backend endpoint that powers Quiqdash v3 must be reachable from an LLM via a single MCP server, with the same auth, the same error semantics, and the same observability (Langfuse evals) as the existing staging-verified tools.

### Out of Scope

- Mastra chat-agent endpoints (`POST /chat/operationsAgent`, `GET /api/memory/threads/{id}`) вЂ” Mastra **is** an LLM agent; wrapping it as another MCP tool would create an agent-calling-agent loop with no value. Documented but not exposed.
- Server-only health endpoints that don't represent agent-useful actions (`GET /api/health`) вЂ” covered by infra monitoring, not an MCP tool surface.
- Building new business endpoints вЂ” this project mirrors what the frontend already calls; it doesn't add new backend capabilities.
- Quiqdash v2 (legacy) endpoints not in the extract вЂ” explicit scope cap.
- A UI for the MCP server вЂ” the existing landing page is sufficient; no dashboard.

## Context

- **Repo:** Brownfield Next.js 16 App Router project; ~32 MCP tools already shipped in `lib/tools/*.ts`. Source-of-truth document for endpoints is `docs/quiqup-api-full-frontend-extract.md` (4.7k lines, generated by scanning `app/hooks/**`, `app/routes/**`, `app/graphql/**` in Quiqdash v3).
- **Auth pattern:** Clerk (`@clerk/nextjs` + `@clerk/mcp-tools`); each tool currently authenticates upstream calls with a Quiqup actor-token derived from the Clerk session.
- **Observability:** Langfuse (`@langfuse/client` + `@langfuse/otel`) is wired; evals live in `evals/` and run via `pnpm eval:*` or CI.
- **Test infra:** Vitest for unit + integration tests, MSW for HTTP mocks, Browserbase + Stagehand for upstream login bootstrap (used to obtain real cookies for staging integration tests).
- **Lessons already on record (from auto-memory):**
  - Quiqup `references` field poisons POST /orders вЂ” use top-level `partner_order_id` instead.
  - Langfuse CLI uses `LANGFUSE_HOST`; SDK uses `LANGFUSE_BASE_URL`; project is on `us.cloud.langfuse.com`.
  - First Langfuse eval showed thin tool descriptions cost 4.5Г— args-consistency вЂ” every new tool needs a rich description with field semantics, not just a one-liner.
- **Branch state:** `feat/update-order-waypoint-tool` is in flight (already committed); the new-project init runs alongside that work.

## Constraints

- **Tech stack:** Next.js 16 (App Router, the new breaking-changes version per `AGENTS.md`), TypeScript, `mcp-handler@^1.1.0`, `@modelcontextprotocol/sdk@^1.29.0`, Zod 4 for schemas. **Read `node_modules/next/dist/docs/` before writing any Next-specific code** (per repo AGENTS.md).
- **Auth boundary:** All upstream calls must go through Clerk в†’ Quiqup actor-token. No tool may bypass this and call upstream APIs anonymously.
- **Multi-host:** 12+ distinct upstream hosts (Platform API, Quiqup REST, Quiqup GraphQL, Orders Core REST + GraphQL, Ex-core, Invoicer, Finance, Audit, Mastra, Supabase, Metabase, Google Places) вЂ” each needs its own `lib/clients/*.ts` and its own env-var family.
- **Description quality:** Every tool description must include field-level semantics, error modes, and the canonical example payload вЂ” thin descriptions are a known regression source.
- **Eval coverage:** Every new service-host family needs at least one Langfuse eval before its tools count as "shipped." CI gate must stay green.
- **Destructive safety:** Batch status transitions and mass operations require explicit confirmation parameters. The agent cannot fire `set_cancelled` on 1000 orders without an explicit `confirm: true` (or equivalent) in the call.
- **PDF/binary responses:** Tools that return PDFs/CSVs return base64 + a content-type hint; no streaming binary directly through MCP transport.

## Key Decisions

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| Source of truth is `docs/quiqup-api-full-frontend-extract.md` | The frontend extract is the most current inventory of what real users do; the BE OpenAPI has gaps (Audit, Invoicer, schema-gap endpoints) that the extract surfaces | вЂ” Pending |
| One MCP tool per endpoint (1:1) for most cases | Predictable, easy to evaluate, matches existing pattern. Higher-level workflows can be composed by agents | вЂ” Pending |
| Phase grouping = service-host families (Platform, Orders Core, Quiqup REST/GraphQL, Ex-core, Invoicer, etc.) | Each host has shared auth/client wiring; grouping minimizes infra churn within a phase | вЂ” Pending |
| Exclude Mastra chat endpoints | Agent-calling-agent loop has no clear value at this stage | вњ“ Good |
| Exclude server-only health route | Not an agent surface | вњ“ Good |
| Multipart + PDF tools return base64 | Keeps the MCP contract uniform across binary/non-binary responses | вЂ” Pending |
| Every new service-host family gets at least one Langfuse eval | Catches regressions before they hit staging | вЂ” Pending |
