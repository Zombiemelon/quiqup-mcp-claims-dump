<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP client → Next.js MCP route | Untrusted inbound. Clerk OAuth at+jwt is the only credential — verified by withMcpAuth in app/[transport]/route.ts before any handler runs. |
| MCP handler → platform-api.quiqup.com | Egress over TLS with `Authorization: Bearer <session-JWT>` minted via the Clerk → Quiqup actor-token bridge (`getQuiqupReadyJwt`). Anonymous calls are impossible — handlers throw when auth.userId is null. |
| Tool args (LLM-supplied) → upstream path/query | The only LLM-controlled values reaching upstream in this plan are `id` (Salesforce id, path param) and `environment` (enum-bounded). No body payloads. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Spoofing | All 7 read handlers | mitigate | Every handler refuses to call upstream when `auth.userId` is null (throws `Error("<tool> requires an authenticated user")`). The Clerk → Quiqup actor-token exchange is the single binding; no caller-supplied identity reaches upstream. |
| T-01-02 | Tampering | get_account_by_id path injection | mitigate | `id` is encoded via `encodeURIComponent` before URL interpolation (mirroring lib/tools/whoami-platform.ts pattern for `/me`). Zod `z.string().min(1)` rejects empty strings at the schema layer. |
| T-01-03 | Information Disclosure | get_account_by_id (admin scope) | accept | Endpoint resolves accounts by Salesforce id; the agent's permission to read another account is enforced by the upstream Platform API (role/scope checks on /accounts/{id}). This MCP server does not duplicate that authorization — Platform API is the source of truth and returns 403 if the caller lacks scope. |
| T-01-04 | Repudiation | All write paths | n/a | This plan has no writes. The 7 tools are read-only GETs. |
| T-01-05 | Denial of Service | get_quiqdash_init repeated calls | accept | Read-only GET on platform-api; rate-limiting is upstream's concern. Agents are advised in the tool description to cache per session. No guardrails applied because the upstream payload is small and the existing M3 read tools follow the same un-rate-limited pattern. |
| T-01-06 | Information Disclosure | Bearer token in logs | mitigate | The audit middleware (lib/middleware/audit.ts) redacts `bearer`/`token`/`jwt` keys via `ALWAYS_REDACT_KEYS`. None of the new tools log the JWT; QuiqupHttpError carries only status + body. |
| T-01-07 | Elevation of Privilege | x-api-version header omission | mitigate | get_permissions explicitly sets `x-api-version: 1` per source-doc line 101 — omitting it on /permissions historically returned a different (legacy) shape. Locked in by grep-gate in Task 1 acceptance criteria. |
| T-01-SC | Tampering | npm/pip/cargo installs | mitigate | RESEARCH.md is intentionally skipped for this phase per the planning_context (research_enabled=false; source-of-truth is docs/quiqup-api-full-frontend-extract.md). No new packages are installed by this plan — only existing deps (zod, msw if already present). If MSW is not in package.json, that single install is gated by a separate human-check task. |
</threat_model>
