## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP client -> Next.js MCP route | Clerk OAuth verified by withMcpAuth before any handler runs. |
| MCP handler -> platform-api.quiqup.com | Bearer JWT via Clerk -> Quiqup actor-token bridge. ALL 5 write handlers use getQuiqupReadyJwt(auth.userId). |
| Tool args (LLM-supplied) -> upstream body | LLM-controlled values: account profile fields (name, emails, currency, bank details), feature names, return-settings fields, team-member email/role. |
| auth.userId -> decide_feature_flags_bulk Identifier | The Identifier sent to upstream is derived server-side from the Clerk subject. NOT LLM-controlled. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-16 | Spoofing | All 5 write handlers | mitigate | Every handler throws on `!auth.userId`. Single binding via getQuiqupReadyJwt; no caller-supplied identity reaches upstream. Locked in by the test-suite's "missing auth.userId throws" assertion in each describe. |
| T-01-17 | Tampering | update_account broad payload abuse | mitigate | All fields are z.string().optional() at the schema layer; settings is z.record (open) — that is the deliberate match for the upstream's open `settings` blob. Bank-detail fields are present here per the source-doc reality (PUT /accounts is the shared endpoint); FIN-05 in Phase 10 will provide the narrower variant that locks bank-only updates. Description warns agents to prefer FIN-05 for bank-only updates. |
| T-01-18 | Elevation of Privilege | decide_feature_flags_bulk Identifier spoof | mitigate | Identifier sourced from `auth.userId` ONLY — `spec.inputSchema.shape.Identifier` is asserted undefined by the test suite. Even if the LLM tries to pass an Identifier in args, it is ignored because the handler builds the body from auth, not args. This is the must-have "decide_feature_flags_bulk binds the input Identifier to the caller's session". |
| T-01-19 | Tampering | return-settings path injection | mitigate | account_id path param goes through encodeURIComponent; Zod enforces `z.string().min(1)`. Default value "me" lets the agent operate on the signed-in account without supplying an id at all. |
| T-01-20 | Elevation of Privilege | create_account_team_member abuse | mitigate | Adding a team member grants Clerk org access to that user. The tool description explicitly warns the agent to confirm intent. role is a free-text string (upstream enforces the allowed enum) — z.string().min(1). The destructive-policy gate from PROJECT.md ("destructive endpoints require confirm:true") was reviewed and judged NOT to apply here: creating a single team member is reversible (you can remove the member upstream) and the destructive-gating policy in PROJECT.md is scoped to batch operations + DELETEs. The plan explicitly accepts this read of the policy; if the executor disagrees, escalate to the user before adding a confirm flag. |
| T-01-21 | Information Disclosure | Bank-detail leakage in logs | mitigate | The audit middleware (lib/middleware/audit.ts) redacts based on key names; bank_iban / bank_swift / bank_account_number are commonly recognized PII-ish fields. Verify (in Task 3 acceptance) that pii-redact patterns cover these or add them — if the existing ALWAYS_REDACT_KEYS list does not, log a follow-up TODO in the SUMMARY rather than blocking the plan (audit-redaction expansion is a cross-cutting concern, not scoped here). |
| T-01-22 | Repudiation | Write traces in audit log | mitigate | Existing audit middleware emits structured records on every tool call. All 5 writes are now subject to that (the registerTool wrapper auto-audits when guardrails are set; if guardrails is unset audit is OFF — this plan does NOT set guardrails because M6 has not landed). The audit-coverage gap is a known M6 deferral, not introduced by this plan; flag in SUMMARY. |
| T-01-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages installed by this plan; uses only existing zod/msw. RESEARCH.md is intentionally absent for this phase (research_enabled=false). |
