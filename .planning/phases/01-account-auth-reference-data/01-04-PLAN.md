<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI runner -> Anthropic API | EVAL runs send dataset prompts to Anthropic; no production data, no user PII — only the hand-authored merchant prompts in the dataset files. |
| CI runner -> Langfuse | Trace+score upload via LANGFUSE_PUBLIC_KEY/SECRET_KEY (CI secrets). |
| Eval runner -> nothing else | Offline: does NOT hit Quiqup or Google Places. The whole eval validates description quality + args-overlap via the LLM, not live upstream traffic. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-23 | Information Disclosure | Dataset prompts leaking PII into Langfuse | mitigate | Datasets are hand-authored merchant questions (e.g. "show me my account profile"); no real customer data, no real Salesforce ids. The single dataset item with a Salesforce id uses the placeholder "003ABC123". |
| T-01-24 | Tampering | Eval skipping in CI (silent regression) | mitigate | EVAL_GATE=1 is set explicitly on both new CI steps. Without it, the gate is no-op (local-dev pattern). Task 3 acceptance grep-asserts both gates are present. |
| T-01-25 | Spoofing | Spoofed eval results | accept | The eval runs in CI under the project's existing secrets management. There is no privilege-escalation surface here — eval results never feed back into tool surface. |
| T-01-26 | Information Disclosure | Drift between production tool description and eval-time description | mitigate | Plan 04 Task 1 explicitly imports `spec.description` from the production module rather than copy-pasting the text (the pattern recent-orders.ts had to deal with via a maintenance comment). Drift is structurally impossible because the eval imports the live module. |
| T-01-27 | Elevation of Privilege | description-quality scorer false positives | accept | The static description-quality scorer asserts substrings like "401", endpoint path, and "whoami_platform". A maintainer could trivially game the score by sprinkling those substrings without writing a real description. We accept that risk because the eval-driven-description-improvement memory documents that the larger LLM-loop scorers (args-overlap) are the real quality signal; description-quality is the cheap lockdown for known disambiguation language. |
| T-01-28 | Spoofing | Eval bypass via the auth-isolation source-string scorer | mitigate | The auth-isolation scorer (Task 2 Step B item 5) literally readFile()'s the implementation files and substring-checks for `getQuiqupReadyJwt`. A maintainer cannot land a regression on lookup_google_place's auth surface without simultaneously deleting or relocating that scorer — and either action shows up in PR review. |
| T-01-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages required if zod-to-json-schema is already a dep. If absent, the executor falls back to inline JSON Schema (Task 1 Step C TODO), keeping this plan zero-install. |
</threat_model>
