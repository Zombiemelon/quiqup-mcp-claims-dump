Auto-throwing would discard data the caller may still want. Tools surface errors[] explicitly in their text output so the LLM can see them."

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP client → Next.js MCP route | Untrusted inbound. Clerk OAuth at+jwt is the only credential — verified by withMcpAuth in app/[transport]/route.ts before any handler runs. |
| MCP handler → orders-api.quiqup.com/graph | Egress over TLS with `Authorization: Bearer <session-JWT>` minted via the Clerk → Quiqup actor-token bridge (`getQuiqupReadyJwt`). Anonymous calls are impossible — handlers throw when auth.userId is null. |
| Tool args (LLM-supplied) → upstream GraphQL variables | LLM-controlled values reaching upstream: `where` (passthrough object), `client_order_ids` (capped at 200), pagination cursors (opaque strings), `orderBy.field` (literal-locked). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing | lookup_orders_ids / bulk_orders_lookup handlers | mitigate | Each handler refuses to call upstream when `auth.userId` is null (throws `Error("<tool> requires an authenticated user")`). The Clerk → Quiqup actor-token exchange is the single binding; no caller-supplied identity reaches upstream. Locked in by "rejects unauthenticated callers" test in each describe. |
| T-03-02 | Tampering | `where` passthrough on Orders Core GraphQL | accept | `where` is `z.object({}).passthrough()` — Quiqup BE validates the OrderWhereInput schema upstream. Tightening client-side would lock out fields the FE quietly uses (over-constraining was the BL-01 footgun pattern in Phase 2). The upstream returns a structured GraphQL error on bad fields; the tool surfaces it via the errors[] contract. |
| T-03-03 | Tampering | `client_order_ids` mass-fetch | mitigate | `z.array(z.number().int().positive()).min(1).max(200)` — caps the request at the upstream's documented `first: 200` hard-cap. Negative or zero IDs are rejected at the schema layer; non-int values are rejected. |
| T-03-04 | Tampering | `orderBy.field` injection | mitigate | `z.literal("SUBMITTED_AT")` — the frontend hard-codes this; allowing free strings would let an LLM probe undocumented sort fields. If Quiqdash extends the enum in future, this widens with explicit review. |
| T-03-05 | Information Disclosure | GraphQL errors[] leakage | accept | The Orders Core errors are upstream validation messages; surfacing them to the agent is the documented partial-failure contract. The error body itself does not echo the JWT or any caller-side secrets. |
| T-03-06 | Information Disclosure | Bearer token in logs | mitigate | OrdersCoreGraphQLClient logs nothing; QuiqupHttpError carries only status + body. The audit middleware (lib/middleware/pii-redact.ts ALWAYS_REDACT_KEYS) redacts `bearer`/`token`/`jwt` keys if any handler arg ever contains one — belt-and-braces. None of these tools accept a token field. |
| T-03-07 | Denial of Service | Repeated paginated scans | accept | Both tools are read-only and no rate-limit is set (matches the Phase 1/2 read convention). The 500-page cap on lookup_orders_ids and 200-id cap on bulk_orders_lookup bound the per-call upstream cost. Quiqup BE applies its own per-account quotas. |
| T-03-08 | Tampering | GraphQL query string injection | mitigate | The `query` string is an inline constant in each tool spec — NEVER built from caller-supplied data. Variables go on the `variables` envelope, parsed by the upstream GraphQL engine; the client serializes via `JSON.stringify` (no string concatenation). |
